## 一、用大白话说清楚这个项目在干什么

### 1.1 一句话版本

**老师给了你一个模拟电商网站（带24个月的经营数据），你要扮演数据分析师，用Python分析这个电商哪里做得好、哪里有问题，然后把分析结果做成一个老板能看懂的仪表盘，并给出具体的经营建议。**

### 1.2 三句话版本

1. **你有一个现成的电商系统**：`client/` 是商城前台（React），`server/` 是商城后台（Node.js），里面有一个 SQLite 数据库，存放了 20,000 个用户、100,000 个订单、700,000 条用户行为记录的模拟数据。

2. **你要新建一个分析系统**：在 `analytics_dashboard/` 目录下用 Python + FastAPI 搭建一个数据分析后台，从 SQLite 数据库中读取经营数据，用数据挖掘的方法（分类、聚类、关联规则、时间序列预测等）进行分析。

3. **最终交付一个能用的仪表盘**：前端页面展示经营指标（GMV、订单数、客单价、转化率等）、数据挖掘结果（用户分群、商品推荐、销售预测等）、经营决策建议，并且用 Git 管理代码，用 Docker Compose 部署到 Ubuntu 服务器。

### 1.3 更直白的类比

想象你在经营一家淘宝店。每天有人浏览商品、有人加入购物车、有人下单、有人退款。你积累了两年的经营数据。现在老板问你：

- 这个店经营得怎么样？哪里赚钱哪里亏钱？
- 哪些用户是忠实客户？哪些快流失了？
- 广告费花得值不值？
- 下个月应该多备哪些货？
- 什么商品应该捆绑销售？

你的任务就是用代码回答这些问题，并做成一个老板每天都能看的仪表盘。

## 三、老师为什么给你这个文件夹结构

这个项目的设计思路是**模拟真实企业的数据分析团队工作流程**：

1. **先有业务系统**（商城 client + server）→ 产生业务数据
2. **再建分析系统**（analytics_dashboard，你要新建的）→ 从数据中挖掘洞察
3. **数据有两个入口**：
   - 方式一：连接 SQLite 数据库（`server/data/eshop.sqlite`）
   - 方式二：读取 CSV 导出文件（`exports/*.csv`）
   - 方式三（教师内网）：通过 `/api/etl/*` 只读接口访问（你目前访问不到）
4. **这就是混合编程的真实场景**：旧系统是 Node.js，新分析服务用 Python，两者通过数据库协作

这也对应了两门课的考核点：
- **IT 项目管理**：Git 版本管理、Docker Compose 部署、项目规划与分工
- **数据挖掘**：分类、聚类、关联规则、时间序列、特征工程等方法在实际业务中的应用

### 5.2 课程中涉及的数据挖掘方法清单

从你的课程 notebooks 和各章讲稿中，你学了以下方法：

**数据预处理与探索**
- 数据读取：SQLite 直连、CSV 导入、ETL API 调用
- 数据质量检查：缺失值、重复值、异常值、值域检查
- 特征工程：RFM 计算、时间窗口、行为特征、缺失值处理、One-hot 编码、StandardScaler
- 可视化：matplotlib/seaborn 趋势图、柱状图、热力图、漏斗图

**分类算法（第4章）**
- 逻辑回归、决策树、随机森林、梯度提升（任选至少3个对比）
- 评估指标：混淆矩阵、Precision、Recall、F1、AUC
- 时间切分（观察窗口 vs 预测窗口，避免数据泄漏）
- 业务阈值选择：触达成本 vs 预期收益

**聚类分析（第5章）**
- K-Means（必做）、层次聚类或 DBSCAN（可选）
- 标准化、距离度量、肘部法、轮廓系数
- 聚类结果命名与运营策略绑定

**关联规则（第6章）**
- Apriori 算法
- 支持度、置信度、提升度
- 规则业务评估：区分"数学上好"与"业务上可用"

**时间序列（第7章）**
- 移动平均、指数平滑
- ADF/KPSS 平稳性检验、ACF/PACF 分析
- ARIMA 或 Prophet 预测
- 评估：MAE、MAPE、RMSE
- 安全库存计算

**营销归因（第8章）**
- CTR、CVR、CPC、CPA、ROAS
- 活动对照组分析（has_control_group）
- 预算迁移建议

### 7.3 多维度切片设计

对于每个分析模块，支持以下维度的数据切片，使分析更有针对性：

| 切片维度 | 可选值 | 应用于哪些分析 |
|---------|-------|--------------|
| **时间维度** | 日/周/月/季度、24个月中任意区间 | 所有分析 |
| **渠道维度** | organic/search/social/email/affiliate/live | 流量、转化、订单、ROAS |
| **用户分层** | 会员等级(normal/plus/vip)、生命周期(high_value/active/new/at_risk/sleeping/coupon_sensitive) | RFM、复购预测、分群 |
| **商品维度** | 12个类目、品牌、价格带(low/mid/high) | 品类贡献、关联规则、库存 |
| **地区维度** | 12个省份 | 用户分布、物流履约 |
| **设备维度** | mobile/desktop/tablet | 流量漏斗、转化率 |
| **活动维度** | 具体活动ID、活动类型(coupon/member/live/new_product/clearance) | 活动效果、ROAS |
| **用户行为维度** | 浏览/搜索/加购/结算/支付 | 漏斗分析、路径诊断 |
| **时间特征维度** | 是否周末、是否活动期、是否节假日 | 季节性分析、预测 |
| **履约维度** | 承运商、是否延迟 | 售后分析、退款影响 |

这些切片通过前端筛选器实现，后端 API 接受筛选参数后返回对应的分析结果。

```
方法：描述性统计、月度趋势、渠道拆解、漏斗分析
输出：
  - 6个KPI卡片：GMV、订单数、买家数、客单价、毛利、退款率
  - 月度GMV趋势图（折线图）
  - 渠道贡献占比（堆叠柱状图/饼图）
  - 流量漏斗（5步漏斗图）
  - 类目销售贡献（柱状图）
管理含义：快速定位经营健康度，回答"这个月表现怎么样？哪里出了问题？"
```

#### 子项目2：RFM与用户宽表 (feature_engineering)

```
输入：dim_user, fact_order, fact_traffic, fact_coupon_use
方法：RFM计算（recency天、frequency次、monetary元）、时间窗口特征、行为计数
输出：
  - 用户宽表（每个用户一行，包含RFM值、行为特征、渠道偏好）
  - R/F/M 分布直方图
  - RFM 分层标签（如：高价值/活跃/潜力/沉睡/流失风险）
管理含义：为复购预测和客户分群提供统一特征底座
```

#### 子项目3：复购预测 (repurchase_prediction)

```
输入：用户宽表（从子项目2获取）
方法：观察窗口(前18个月)→预测窗口(后6个月)，对比逻辑回归、随机森林、梯度提升
      时间切分训练/测试集，Precision/Recall/F1/AUC评估
输出：
  - 3个模型效果对比表
  - 特征重要性排名
  - 高潜复购用户名单（Top N）
  - 不同阈值下的触达成本与预期ROI
  - 模型解释（SHAP或特征重要性图）
管理含义：控制营销成本，精准触达最可能复购的用户
```

#### 子项目4：客户分群 (customer_clustering)

```
输入：RFM特征（标准化后的 R/F/M 值）
方法：K-Means聚类，肘部法+轮廓系数确定K值，层次聚类辅助验证
输出：
  - 聚类散点图（R vs F, F vs M）
  - 每个簇的画像：规模、平均R/F/M、主要渠道、偏好类目
  - 簇命名：如"高价值VIP"、"活跃复购客"、"沉睡待唤醒"、"价格敏感客"、"流失边缘"
  - 每类用户的运营策略（发券类型、触达频率、推荐商品）
管理含义：不同用户群体匹配不同运营策略，避免"一刀切"
```

#### 子项目5：关联规则 (association_rules)

```
输入：fact_order_item, dim_product
方法：按订单聚合SKU构建购物篮 → Apriori算法挖掘频繁项集 → 计算支持度/置信度/提升度
输出：
  - Top 20 关联规则表（按提升度排序）
  - 商品组合推荐（捆绑销售候选）
  - 凑单推荐候选（低价高毛利商品）
  - 标注"数学好但业务不可用"的规则（如提升度高但支持度极低）
管理含义：组合销售、推荐位实验、客单价提升
```

#### 子项目6：销售预测 (sales_forecast)

```
输入：daily_business_summary, fact_order_item, dim_product
方法：日GMV聚合 → ADF平稳性检验 → ACF/PACF分析 → ARIMA建模
      同时提供30天移动平均基线 → 计算波动系数 → 安全库存建议
输出：
  - 未来30天GMV预测（含置信区间）
  - Top 5 品类销量预测
  - 安全库存金额建议
  - 预测误差指标（MAE、MAPE、RMSE）
  - 季节性分解图（趋势+季节+残差）
管理含义：备货计划、库存风险控制、活动期提前备货
```

#### 子项目7：营销归因 (marketing_attribution)

```
输入：fact_ads_spend, dim_campaign, fact_order
方法：计算CTR/CVR/CPA/ROAS → 渠道效率对比 → 有对照组的活动做增量分析
输出：
  - 各渠道ROAS排行榜
  - 广告花费 vs 转化散点图
  - 渠道CPA对比
  - 预算迁移建议（哪些渠道加投、哪些压缩）
  - 对照组vs活动组 GMV 对比
管理含义：优化广告预算分配，识别真正有效的投放渠道
```

#### 子项目8：综合决策中心 (decision_board)

```
输入：前7个子项目的分析结果
方法：综合评分、问题优先级排序、建议生成
输出：
  - 本周经营健康评分
  - P0/P1/P2 优先级决策建议（每条含数据证据、预期收益、风险）
  - 行动清单：用户触达、商品策略、渠道调整、库存补货
  - Git 项目路线与里程碑
管理含义：用于最终答辩和管理层汇报
```

## 九、数据读取的三种方式

老师提到由于 ETL 接口在内网无法直接访问，你有两种数据获取方式：

### 方式1：直连 SQLite（推荐）

```python
import sqlite3
db_path = "eshop-dashboard-practice/server/data/eshop.sqlite"
conn = sqlite3.connect(f"file:{db_path}?mode=ro", uri=True)  # 只读模式
df = pd.read_sql_query("SELECT * FROM fact_order LIMIT 100", conn)
```

优点：数据完整、查询灵活、不需要单独部署
缺点：需要了解表结构

### 方式2：读取 CSV 导出文件

```python
import pandas as pd
df = pd.read_csv("eshop-dashboard-practice/exports/fact_order.csv")
```

优点：最简单、不需要 SQLite 连接
缺点：数据是静态快照、无法做灵活查询

### 方式3：部署商城后通过 ETL API（加分项）

把 server/ 部署起来后，ETL API 就运行在 `localhost:38173/api/etl/*`，可以直接用 HTTP 请求获取数据。这就是老师说的"第一种方式"——在自己电脑上部署商城。

```bash
npm run start:mall-api   # 启动商城 API（port 38173）
# 然后访问 http://localhost:38173/api/etl/metrics 就能拿到指标
```

## 十、关键提醒

1. **不要修改商城源码**（client/ 和 server/ 是业务系统，保留原样）。你的代码全部写在 `analytics_dashboard/` 里。

2. **每个指标都要能说清楚口径**。比如 GMV = `SUM(paid_amount) WHERE status IN ('paid', 'completed')`，答辩时老师会追问。

3. **时间切分很重要**。做分类模型时不能用未来数据预测过去，必须用观察窗口→预测窗口的设计。

4. **模型结果要能解释**。不要只输出"模型 AUC 0.85"，要说明"这意味着什么，什么特征最重要，对经营有什么帮助"。

5. **Git 提交要能体现功能演进**。每完成一个子项目就提交一次，最终 `git log --oneline --graph` 要能看出来项目是怎么一步步做出来的。

6. **Docker Compose 是加分项**。如果本机没有 Docker，至少保证配置文件语法正确，并在 Ubuntu 环境（或 WSL）中验证。

7. **不需要重做商城前台功能**。购物车、下单、支付这些不在你的任务范围内，你的重点是后台数据分析和决策支持。
