# 学生功能需求：后台运营管理与决策支持系统

本文档将课程大纲拆解为学生需要实现的功能需求。学生不需要重做商城前台，应基于 `course_dataset_v2`、分析视图和商城源表完成数据挖掘与经营决策支持。

## 总体要求

学生系统应至少包含：

- 后台登录与角色：管理员、运营、分析师。
- 数据看板：经营总览、筛选器、趋势图、明细钻取。
- 数据挖掘模块：至少 2 个模型型任务。
- 运营动作模块：用户触达、商品策略、活动策略、库存策略或履约治理。
- 导出功能：CSV 或 Excel。
- 审计功能：运营动作写入 `admin_action_logs` 或等价表。
- 口径说明：每个核心指标页面必须展示数据来源和计算口径。

## 原课程 1-7 章对应的学生实现模块

学生系统不是简单做页面，而是把课程知识点转化为可运行功能。最低覆盖关系如下：

| 原课程章节 | 必须体现的知识点 | 对应系统功能 |
|---|---|---|
| 第 1 章 数据挖掘概述 | CRISP-DM、管理决策价值、Python/Jupyter 工具链 | 项目说明页、业务问题定义页、数据挖掘流程说明、notebook 复现入口 |
| 第 2 章 数据准备与探索 | 多源读取、数据质量、EDA、描述性统计、可视化 | R0 数据接入与质量报告、R1 经营驾驶舱 |
| 第 3 章 数据预处理 | 缺失值、异常值、重复值、编码、缩放、特征选择、类别不平衡 | R0 清洗流水线、建模宽表生成、R4 分类建模数据准备 |
| 第 4 章 分类算法 | 逻辑回归、决策树、随机森林、梯度提升、评估指标、业务阈值 | R4 复购/流失/券敏感/退款风险预测 |
| 第 5 章 聚类分析 | K-Means、层次聚类、轮廓系数、客户细分、产品细分 | R5 用户聚类与商品聚类 |
| 第 6 章 关联规则 | Apriori、FP-growth、支持度、置信度、提升度、购物篮 | R6 购物篮分析、组合推荐、交叉销售 |
| 第 7 章 时间序列 | 趋势、季节性、ADF/KPSS、ACF/PACF、ARIMA、预测误差 | R7 GMV/销量/SKU 需求预测 |

鼓励扩展到文本挖掘、异常检测、推荐系统、营销增量评估和综合决策中心。

## R0：数据接入、ETL 与清洗模块

### 目标

让学生理解数据源、清洗规则和分析宽表如何生成。

### 数据来源

- `dim_user`
- `dim_product`
- `dim_campaign`
- `fact_order`
- `fact_order_item`
- `fact_refund`
- `fact_traffic`
- `fact_coupon_use`

### 功能需求

1. 数据源浏览：
   - 展示所有可用 `dim_*` 和 `fact_*` 表。
   - 展示表粒度、主键、核心字段。
   - 至少支持两种读取方式：SQLite 直连与 CSV 导入。
   - 可选支持 API 或 JSON 数据接入，用于体现半结构化数据处理。

2. 数据质量检查：
   - 缺失值统计。
   - 重复主键检查。
   - 数据类型和值域检查。
   - 时间范围和数据新鲜度检查。
   - 金额异常检查，例如订单金额小于 0、订单行金额不等于订单金额的异常。
   - 状态分布检查，例如 `completed`、`paid`、`refunded`。

3. 指标口径配置：
   - GMV。
   - 净销售额。
   - 毛利。
   - 退款率。
   - 转化率。
   - 客单价。

4. 清洗结果导出：
   - 输出经营日表。
   - 输出用户建模宽表。
   - 输出商品建模宽表。

5. 预处理流水线：
   - 缺失值处理：删除、均值/中位数/众数填充，至少比较两种策略。
   - 异常值处理：IQR、Z-score 或 Isolation Forest 至少一种。
   - 类别变量编码：One-hot 或 Ordinal Encoding。
   - 特征缩放：StandardScaler、MinMaxScaler 或 RobustScaler 至少一种。
   - 特征选择：相关性、方差过滤、特征重要性或 PCA 至少一种。

### 验收标准

- 至少展示 5 类数据质量问题或质量指标。
- 至少生成 2 张可复用建模宽表。
- 每个指标必须能追溯到字段和过滤条件。
- 至少有一个可复用的预处理流水线函数或脚本。

## R1：经营驾驶舱

### 目标

实现经营管理最基本的可视化和诊断入口。

### 数据来源

- `daily_business_summary`
- `fact_order`
- `fact_order_item`
- `dim_product`
- `dim_user`

### 功能需求

1. 核心指标卡：
   - GMV。
   - 订单数。
   - 买家数。
   - 客单价。
   - 毛利。
   - 退款率。

2. 多维筛选：
   - 日期范围。
   - 渠道。
   - 类目。
   - 用户分群。
   - 活动。

3. 趋势分析：
   - 日/周/月 GMV。
   - 订单量趋势。
   - 毛利趋势。

4. 结构分析：
   - 类目销售贡献。
   - 渠道贡献。
   - 会员等级贡献。
   - Top / Bottom 商品。

### 决策输出

- 找出增长贡献最大的渠道或类目。
- 找出高 GMV 但低毛利的问题品类。
- 给出本周经营异常摘要。

### 验收标准

- 至少 6 个指标卡。
- 至少 4 个筛选器。
- 至少 4 张图表。
- 至少 1 个明细钻取页面。

## R2：流量漏斗与转化诊断

### 目标

分析从访问到支付的转化过程，定位流失环节。

### 数据来源

- `fact_traffic`
- `fact_order`
- `dim_campaign`
- `dim_product`

### 功能需求

1. 漏斗展示：
   - `view_home`
   - `view_product`
   - `add_to_cart`
   - `checkout`
   - `pay_success`

2. 分组对比：
   - 渠道。
   - 设备。
   - 活动。
   - 类目。

3. 路径诊断：
   - 高访问低加购商品。
   - 高加购低支付商品。
   - 高结算放弃会话。

4. 未转化用户列表：
   - 支持按最近访问时间、加购金额、浏览类目筛选。
   - 可标记为再营销候选人群。

### 决策输出

- 哪些渠道带来低质量流量。
- 哪些商品详情页需要优化。
- 哪些用户适合再营销。

### 验收标准

- 支持至少 3 个维度的漏斗对比。
- 输出一个再营销候选用户列表。
- 能导出未转化会话明细。

## R3：RFM、Cohort 与用户运营

### 目标

### 数据来源

- `dim_user`
- `fact_order`
- `fact_coupon_use`
- `fact_traffic`

### 功能需求

1. RFM 计算：
   - Recency。
   - Frequency。
   - Monetary。
   - RFM 分层标签。

2. Cohort 留存：
   - 按注册月份或首购月份。
   - 观察 1、2、3、6、12 个月留存。

3. 用户画像：
   - 地区。
   - 注册渠道。
   - 偏好类目。
   - 券使用偏好。
   - 最近活跃。

4. 运营动作：
   - 批量打标签。
   - 导出触达名单。
   - 生成优惠券发放建议。

### 决策输出

- 高价值用户维护策略。
- 流失用户唤醒策略。
- 价格敏感用户优惠策略。

### 验收标准

- 至少 1 张 cohort 热力图或等效表。
- 至少 1 个用户名单导出功能。

## R4：分类预测模型

### 目标

用机器学习支持运营决策，至少完成一个分类任务。

### 可选任务

1. 复购预测：预测用户未来 30/60/90 天是否复购。
2. 流失预警：预测用户是否进入沉默或流失。
3. 券敏感度：预测用户收到优惠券后是否核销。
4. 退款风险：预测订单是否更可能退款。

### 数据来源

- `dim_user`
- `fact_order`
- `fact_order_item`
- `fact_coupon_use`
- `fact_traffic`
- `fact_refund`
- `fact_fulfillment`
- `fact_product_review`

### 功能需求

1. 建模数据集生成：
   - 明确观察窗口。
   - 明确标签窗口。
   - 明确训练集、验证集、测试集。

2. 模型训练：
   - 至少比较 3 个模型。
   - 必选模型池：逻辑回归、决策树、随机森林、梯度提升中至少 3 个。
   - 可选进阶：SVM、神经网络、XGBoost 或 LightGBM。

3. 模型评估：
   - 混淆矩阵。
   - Precision。
   - Recall。
   - F1。
   - AUC。
   - 业务收益或成本模拟。

4. 模型结果落地：
   - 输出预测名单。
   - 输出风险分数或机会分数。
   - 支持运营筛选和导出。
   - 支持调整阈值，并展示精确率、召回率、触达成本和预期收益变化。

### 决策输出

- 哪些用户值得触达。
- 哪些订单需要客服或风控提前介入。
- 触达阈值如何影响成本和收益。

### 验收标准

- 至少一个完整分类模型。
- 有时间切分，不能随机打乱导致未来数据泄漏。
- 有模型解释或特征重要性。
- 模型结果能进入后台页面。
- 至少包含一次业务阈值选择说明或 A/B 测试设计。

## R5：聚类分群

### 目标

用无监督学习发现用户或商品群体结构。

### 可选任务

1. 用户聚类：
   - 购买力。
   - 活跃度。
   - 品类偏好。
   - 优惠敏感。
   - 售后体验。

2. 商品聚类：
   - 销量。
   - 毛利。
   - 价格带。
   - 退款率。
   - 评分。
   - 库存周转。

### 功能需求

1. 特征标准化。
2. 聚类数量选择：
   - 肘部法。
   - 轮廓系数。
   - 可选补充 Calinski-Harabasz 或 Davies-Bouldin 指数。
3. 聚类结果解释：
   - 每类用户或商品命名。
   - 展示各簇特征均值。
   - 展示各簇规模、销售贡献、毛利贡献或退款风险。
4. 策略绑定：
   - 用户簇绑定运营策略。
   - 商品簇绑定经营策略。
5. 算法比较：
   - 至少使用 K-Means。
   - 可选比较层次聚类或 DBSCAN。

### 决策输出

- 哪些用户适合优惠，哪些适合会员权益。
- 哪些商品是利润款、引流款、风险款或清仓款。

### 验收标准

- 至少完成一种聚类。
- 至少 3 个可解释簇。
- 每个簇必须有业务命名和行动建议。

## R6：关联规则与推荐

### 目标

从订单行中发现商品搭配关系，用于捆绑、凑单和交叉销售。

### 数据来源

- `fact_order_item`
- `dim_product`

### 功能需求

1. 购物篮构建：
   - 按订单聚合 SKU 或类目。
   - 支持按类目、价格带、时间段筛选。

2. 规则挖掘：
   - Apriori。
   - 可选 FP-growth。
   - 支持度。
   - 置信度。
   - 提升度。

3. 推荐展示：
   - 商品 A 推荐商品 B。
   - 组合销售建议。
   - 满减凑单建议。

4. 策略模拟：
   - 根据客单价和毛利估算组合销售收益。

### 决策输出

- 哪些商品适合捆绑。
- 哪些商品适合出现在购物车推荐位。
- 哪些低价高毛利商品适合凑单。

### 验收标准

- 至少输出 20 条规则。
- 每条规则包含支持度、置信度、提升度。
- 至少 5 条规则转化为运营建议。
- 至少说明一条“数学指标高但业务不适合上线”的规则。

## R7：时间序列与需求预测

### 目标

预测 GMV、订单量、品类销量或 SKU 需求，为预算和库存决策服务。

### 数据来源

- `dim_date`
- `fact_order`
- `fact_order_item`
- `fact_ads_spend`
- `fact_inventory_movement`
- `dim_campaign`

### 功能需求

1. 时间序列构建：
   - 日 GMV。
   - 日订单量。
   - 品类销量。
   - SKU 销量。

2. 季节性分析：
   - 周末效应。
   - 月度趋势。
   - 活动冲击。
   - 趋势、季节性和随机项分解。

3. 预测模型：
   - 移动平均或指数平滑。
   - ARIMA、Prophet 或其他时间序列模型。
   - 至少比较一个基线模型。
   - 至少对一个序列完成 ADF 或 KPSS 平稳性检验。
   - 至少对一个序列展示 ACF/PACF 或等价的滞后相关分析。

4. 预测结果：
   - 未来 7/14/30 天预测。
   - 置信区间或误差范围。
   - 预测误差指标。

### 决策输出

- 补货建议。
- 活动期备货建议。
- 销售目标预警。

### 验收标准

- 至少一个时间序列预测任务。
- 使用时间切分验证。
- 输出 MAE、MAPE 或 RMSE。
- 至少 5 个 SKU 或 3 个品类给出补货建议。
- 明确说明预测误差对库存成本或缺货风险的影响。

## R8：营销活动与增量评估

### 目标

评估活动是否真正带来增量，而不是补贴原本会购买的人。

### 数据来源

- `dim_campaign`
- `fact_ads_spend`
- `fact_order`
- `fact_coupon_use`
- `fact_traffic`

### 功能需求

1. 活动看板：
   - 曝光。
   - 点击。
   - CTR。
   - 转化。
   - GMV。
   - 折扣成本。
   - 广告消耗。
   - ROAS。

2. 优惠券分析：
   - 发放数。
   - 核销数。
   - 核销率。
   - 券带来的 GMV。
   - 券成本。

3. 增量评估：
   - 对 `has_control_group = 1` 的活动进行活动组/对照组比较。
   - 对无对照组活动做活动前后对比。

4. 预算建议：
   - 渠道预算迁移建议。
   - 活动停投或加投建议。

### 决策输出

- 哪些活动应该继续。
- 哪些活动应该降低优惠力度。
- 哪些渠道应增加预算。

### 验收标准

- 至少分析 5 个活动。
- 至少 1 个活动使用对照组或准对照思路。
- 输出预算调整建议。

## R9：履约、售后与体验治理

### 目标

把物流、退款和评价纳入经营决策。

### 数据来源

- `fact_fulfillment`
- `fact_refund`
- `fact_product_review`
- `fact_order`
- `dim_product`

### 功能需求

1. 履约分析：
   - 承运商延迟率。
   - 省份延迟率。
   - 品类延迟率。

2. 退款分析：
   - 退款率。
   - 退款金额。
   - 退款原因。
   - 商品/供应商退款排行。

3. 评论分析：
   - 平均评分。
   - 差评率。
   - 内容标签分布。

4. 风险清单：
   - 高退款商品。
   - 高延迟地区。
   - 高差评供应商。

### 决策输出

- 供应商治理名单。
- 物流承运商调整建议。
- 商品页面说明或质检优化建议。

### 验收标准

- 至少 3 类风险榜单。
- 至少 1 个履约体验对退款或复购影响的分析。
- 至少 3 条治理建议。

## R10：库存与商品策略

### 目标

把销售、库存、毛利和预测结合，支持选品与补货。

### 数据来源

- `fact_inventory_movement`
- `fact_order_item`
- `dim_product`
- `fact_refund`
- `fact_product_review`

### 功能需求

1. SKU 动销分析：
   - 销量。
   - 销售额。
   - 毛利。
   - 库存流水。
   - 周转速度。

2. ABC 分类：
   - 按销售额。
   - 按毛利。
   - 按销量。

3. 商品策略：
   - 补货。
   - 清仓。
   - 提价/降价。
   - 下架观察。

4. 预警：
   - 缺货风险。

### 决策输出

### 验收标准

- 至少 4 类商品策略标签。
- 至少 20 个 SKU 的策略建议。
- 每条建议必须有数据证据。

## R11：综合决策中心

### 目标

将各类挖掘结果统一成管理层可执行的决策工作台。

### 功能需求

1. 决策摘要：
   - 本周经营健康度。
   - 最大增长机会。
   - 最大经营风险。
   - 推荐动作。

2. 动作管理：
   - 创建运营动作。
   - 绑定用户、商品、活动或供应商。
   - 记录负责人、截止日期、预期收益。
   - 写入审计日志。

3. 效果追踪：
   - 动作前后指标对比。
   - 是否达到预期收益。
   - 复盘结论。

### 验收标准

- 至少集成 3 个分析模块结果。
- 至少支持 1 类运营动作闭环。
- 至少有 1 个动作效果复盘页面。
