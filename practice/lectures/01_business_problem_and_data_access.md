# 第 1 章 商业问题定义与 ETL 数据接入

## 1. 本章导读：先听懂老板的问题，再打开数据表

很多同学第一次接触“数据挖掘”时，会以为这门课的重点是算法：分类、聚类、关联规则、时间序列、随机森林、XGBoost。算法当然重要，但如果我们在商业场景中使用数据挖掘，第一步绝不是选算法，而是听懂问题。

真实企业里的管理者很少会这样提问：

> 请帮我训练一个随机森林模型。

他们更可能这样说：

> 上个月销售额为什么下降？  
> 哪些用户值得发券？  
> 商品为什么有人看但没人买？  
> 下个月要备多少货？  
> 广告预算应该投给搜索、直播，还是社交媒体？

这些问题不是数学题，而是经营题。数据分析师的工作，就是把经营题翻译成可以取数、可以计算、可以验证、可以执行的分析任务。

本章要做的事情，就是带你完成这次翻译。我们会认识商城经营中最基础的一组概念：GMV、净销售额、毛利、订单数、买家数、客单价、流量、流量漏斗、转化率；也会认识本课程的数据入口：ETL 接口；还会学习如何查看数据表、字段结构和数据质量报告。

你可以把本章看作进入数据挖掘世界之前的一张地图。没有这张地图，后面即使模型跑出来了，也很可能不知道它能帮助谁、解决什么问题、创造什么价值。

## 3. 商业故事：周一早会上的三个问题

想象你是 Course eShop 商城新入职的数据分析师。周一早会，CEO 看着经营看板说：

> “上个月 GMV 环比下降了。到底是流量不行，还是转化不行？”

运营负责人接着说：

> “我们有 2 万注册用户，但不知道哪些用户值得发券。发多了浪费，发少了没效果。”

供应链负责人也提出问题：

> “下个月有促销活动，要备多少货？如果备少了会缺货，备多了又压库存。”

这些问题看起来不同，但它们有共同点：都不能靠感觉回答。我们需要数据。

不过，拿到数据之前，我们先要弄清楚这些词是什么意思。比如 CEO 说 GMV 下降，如果你连 GMV 是什么都没搞清楚，就不可能判断下降原因。

## 4. 商业概念 1：GMV 是什么

GMV 是 Gross Merchandise Volume 的缩写，通常翻译为“商品交易总额”或“成交总额”。在电商业务中，它表示在某段时间内，平台产生的交易规模。

一个简单例子：

假设今天商城有 3 笔支付成功订单：

| 订单 | 支付金额 |
|---|---:|
| A | 100 元 |
| B | 200 元 |
| C | 300 元 |

那么今天 GMV 可以简单理解为：

```text
GMV = 100 + 200 + 300 = 600 元
```

但在真实业务中，GMV 一定要讲清楚口径。本课程 ETL 接口中的 GMV 口径是：

```text
SUM(orders.paid_amount)
WHERE status IN ('paid', 'completed')
```

也就是说，它统计的是状态为已支付或已完成订单的支付金额总和。

### 4.1 GMV 不等于收入

GMV 代表交易规模，但不一定等于企业最终收入。

假设用户支付了 1000 元，后来退款 200 元。那么从交易规模看，曾经发生过 1000 元交易；但从最终销售结果看，企业真正留下来的可能只有 800 元。

因此我们还需要“净销售额”。

### 4.2 GMV 不等于利润

利润要考虑成本。

假设一件商品卖 100 元，进货成本是 60 元。即使 GMV 是 100 元，毛利也只有：

```text
毛利 = 售价 - 成本 = 100 - 60 = 40 元
```

如果为了冲 GMV，大量发优惠券，用户支付金额可能上升，但毛利不一定上升。甚至可能出现 GMV 很漂亮、利润很难看的情况。

所以管理者不能只看 GMV。GMV 是规模指标，不是盈利指标。

### 4.3 GMV 的经济含义

GMV 回答的问题是：

> 平台上发生了多大规模的交易？

它适合衡量业务体量、市场活跃度、销售规模。但它不能单独回答：

- 公司赚了多少钱？
- 用户是否满意？
- 活动是否划算？
- 商品毛利是否健康？

这些问题需要结合净销售额、毛利、退款率、复购率、ROI 等指标。

## 5. 商业概念 2：净销售额、毛利和退款

### 5.1 净销售额

净销售额可以简单理解为：

```text
净销售额 = GMV - 退款金额
```

如果 GMV 是 100 万，退款 8 万，那么净销售额约为 92 万。

净销售额比 GMV 更接近“最终留下来的销售成果”。如果一个商城 GMV 增长很快，但退款也增长很快，就要警惕：增长可能质量不好。

### 5.2 毛利

毛利可以简单理解为：

```text
毛利 = 销售收入 - 商品成本 - 优惠折扣
```

在订单行层面，可以近似写成：

```text
毛利 = (销售单价 - 成本单价) × 数量 - 优惠金额
```

毛利回答的问题是：

> 卖出去这些商品后，扣掉商品成本，还剩多少经营空间？

如果一个活动带来很多 GMV，但毛利很低，说明它可能只是“花钱买规模”。这种活动不一定值得长期做。

### 5.3 退款率

退款率可以从两个角度理解：

```text
退款订单率 = 退款订单数 / 支付订单数
退款金额率 = 退款金额 / GMV
```

退款率高可能说明：

- 商品质量有问题
- 商品描述和实物不一致
- 物流履约慢
- 售后体验差
- 活动吸引了不合适用户

所以退款不是财务小事，而是经营质量信号。

## 6. 商业概念 3：订单数、买家数和客单价

### 6.1 订单数

订单数是某段时间内产生了多少笔订单。注意，一位用户可以产生多笔订单。

订单数回答：

> 用户一共买了多少次？

### 6.2 买家数

买家数是某段时间内有多少个不同用户购买了商品。它通常按 `user_id` 去重。

买家数回答：

> 有多少不同的人参与了购买？

如果订单数增长，但买家数不增长，可能是老用户买得更频繁；如果买家数增长，说明购买人群扩大。

### 6.3 客单价

客单价是平均每笔订单的金额：

```text
客单价 = GMV / 订单数
```

假设 GMV 是 100 万，订单数是 1000 单，那么客单价是 1000 元。

客单价回答：

> 平均每笔订单买多少钱？

客单价升高可能是好事，也可能不是。好的一面是用户买得更多或买得更贵；不好的一面可能是低价用户流失，只剩少数高价订单。因此客单价必须和订单数、买家数一起看。

## 7. 商业概念 4：流量是什么

电商里的“流量”，不是马路上的车流，而是用户访问、浏览、点击、搜索、加购等行为的总称。

一个用户打开首页，是一次访问行为；点击商品页，是一次商品浏览；加入购物车，是一次加购行为；进入结算，是一次结算行为；支付成功，是一次购买行为。

在本课程数据中，这些行为记录在 `fact_traffic` 或源表 `page_events` 中，字段 `event_type` 表示行为类型。

常见事件包括：

| event_type | 含义 |
|---|---|
| view_home | 浏览首页 |
| view_product | 浏览商品页 |
| add_to_cart | 加入购物车 |
| checkout | 进入结算 |
| pay_success | 支付成功 |

流量回答的问题是：

> 用户有没有来？来了以后做了什么？

销售额下降时，流量是最先要看的原因之一。如果用户根本没来，后面的商品页、加购、支付都无从谈起。

## 8. 商业概念 5：什么叫流量漏斗

“流量漏斗”是电商分析中非常重要的概念。它描述用户从进入商城到最终购买的逐步流失过程。

为什么叫漏斗？因为每经过一步，用户数量通常会减少，就像水从漏斗上方流到下方，越往下越少。

一个典型电商漏斗是：

```text
首页浏览 -> 商品页浏览 -> 加入购物车 -> 进入结算 -> 支付成功
```

假设有 10000 人浏览首页：

| 阶段 | 人数 |
|---|---:|
| 首页浏览 | 10000 |
| 商品页浏览 | 8000 |
| 加入购物车 | 4000 |
| 进入结算 | 2500 |
| 支付成功 | 2000 |

这说明从首页到商品页流失 2000 人，从商品页到加购流失 4000 人，从加购到结算流失 1500 人，从结算到支付流失 500 人。

### 8.1 整体转化率

整体转化率是从最开始到最终目标的比例：

```text
整体支付转化率 = 支付成功人数 / 首页浏览人数
```

在上面的例子中：

```text
整体支付转化率 = 2000 / 10000 = 20%
```

### 8.2 环节转化率

环节转化率是相邻两步之间的比例：

```text
首页到商品页转化率 = 商品页浏览人数 / 首页浏览人数
商品页到加购转化率 = 加购人数 / 商品页浏览人数
加购到结算转化率 = 结算人数 / 加购人数
结算到支付转化率 = 支付人数 / 结算人数
```

环节转化率告诉我们问题发生在哪一步。

### 8.3 流量漏斗的经济含义

流量漏斗不是一张漂亮图，它直接对应经营动作。

如果首页到商品页流失严重，可能说明：

- 首页推荐不吸引人
- 首屏信息不清楚
- 用户找不到想买的商品
- 渠道来的用户和页面内容不匹配

如果商品页到加购流失严重，可能说明：

- 价格缺乏吸引力
- 图片和描述不够好
- 评论和评分影响信任
- 商品规格复杂

如果加购到结算流失严重，可能说明：

- 运费太高
- 优惠门槛不合适
- 用户还在比价
- 购物车缺少凑单提示

如果结算到支付流失严重，可能说明：

- 支付方式不方便
- 页面加载慢
- 用户担心安全
- 地址或配送信息有问题

所以流量漏斗的核心价值是：

> 找到用户在哪一步流失，然后决定优化哪个环节。

## 9. 数据概念 1：ETL 接口是什么

ETL 是 Extract、Transform、Load 的缩写。

- Extract：抽取，把数据从业务系统中取出来。
- Transform：转换，把脏数据、不同格式数据整理成可分析结构。
- Load：加载，把整理后的数据放到分析系统或数据仓库。

本课程中的 ETL 接口，就是已经为教学准备好的只读数据服务。它不让我们直接修改数据库，而是通过 HTTP 接口读取数据。

为什么这很重要？

第一，安全。学生不会误删数据。

第二，统一。所有人使用同样口径。

第三，可复现。Notebook 每次运行都能从同一个入口取数。

第四，接近真实工作。真实企业中，分析师通常也通过数据仓库、API、BI 系统或权限控制后的查询平台获取数据。

## 10. 数据概念 2：维度表和事实表

在本课程中，你会看到两类表：

### 10.1 维度表

维度表描述对象。它回答“是谁”“是什么”“属于哪里”。

例如：

- `dim_user`：用户是谁，来自哪里，会员等级是什么。
- `dim_product`：商品是什么，属于哪个类目，价格和成本是多少。
- `dim_campaign`：活动是什么，渠道是什么，预算是多少。

维度表像名片。

### 10.2 事实表

事实表记录事件。它回答“发生了什么”“发生了多少”“什么时候发生”。

例如：

- `fact_order`：发生了订单。
- `fact_order_item`：订单里买了哪些商品。
- `fact_traffic`：用户发生了哪些行为。
- `fact_ads_spend`：广告每天花了多少钱。

事实表像流水账。

一句话记忆：

```text
维度表描述对象，事实表记录事件。
```

## 11. 数据概念 3：数据质量

数据质量是指数据是否可信、完整、一致、合理。

常见数据质量问题包括：

- 缺失值：用户 ID 为空，订单日期为空。
- 重复值：同一个订单出现两次。
- 异常值：支付金额为负数，年龄为 300 岁。
- 逻辑错误：退款金额大于支付金额。
- 时间错误：支付时间早于下单时间。

为什么数据质量要在建模前检查？

因为模型不会判断数据是不是荒谬。你给它错误数据，它也会认真学习，并输出看似精确但实际错误的结果。

本课程 `/quality` 接口会返回数据质量检查报告，包含 pass、warn、fail。

- pass：通过。
- warn：有风险，需要关注。
- fail：严重问题，应先处理。

## 16. 补充讲稿：什么叫流量漏斗

同学们，流量漏斗是电商经营中最重要、也最容易被误解的概念之一。

“流量”指进入商城或接触商城内容的用户访问。用户可能来自搜索广告、社交媒体、直播间、自然搜索、短信、老客直接访问等。流量不是订单，它只是生意发生的入口。

“漏斗”是一种比喻。想象一个上宽下窄的漏斗：很多用户从上方进入，但不是所有人都会一路走到底。有人只看了首页就离开，有人看了商品详情但没有加购，有人加购后没有结算，有人结算后没有支付。越往下，留下的人越少。

电商流量漏斗通常可以写成：

```text
进入商城 -> 浏览商品 -> 加入购物车 -> 提交结算 -> 支付成功
```

每一层都有明确的商业含义。

进入商城说明用户被渠道、品牌、活动或需求带到了这里。它回答“有没有人来”。

浏览商品说明用户开始了解商品。它回答“用户是否对商品产生兴趣”。

加入购物车说明用户有较强购买意向。它回答“用户是否把商品纳入选择范围”。

提交结算说明用户已经接近成交。它回答“价格、优惠、库存、配送是否还能支撑购买”。

支付成功才是真正的交易结果。它回答“这次访问是否变成 GMV”。

漏斗分析最重要的不是背诵这些层级，而是学会定位问题。

如果进入商城的人少，问题可能在流量获取：广告预算不足、渠道选择不对、活动吸引力弱、自然搜索排名低。

如果进入商城很多，但浏览商品少，问题可能在首页承接：页面加载慢、导航不清楚、活动入口不明显、用户来错了地方。

如果浏览商品很多，但加购少，问题可能在商品吸引力：价格不合适、图片不清楚、评价不足、卖点没有打动用户。

如果加购很多，但结算少，问题可能在优惠门槛、库存、运费、配送时间或用户犹豫。

如果结算很多，但支付少，问题可能在支付流程、优惠券不可用、价格突然变化、风控拦截或用户最后一刻放弃。

所以，流量漏斗不是一个漂亮图形，而是一套经营诊断方法。它让我们知道 GMV 低到底是“没人来”，还是“来了不看”，还是“看了不买”，还是“快买了又走”。

流量漏斗常用两个转化率。

第一是整体转化率：

```text
整体转化率 = 支付成功人数 / 进入商城人数
```

它告诉我们从入口到成交的总效率。

第二是分步转化率：

```text
浏览转化率 = 浏览商品人数 / 进入商城人数
加购转化率 = 加购人数 / 浏览商品人数
结算转化率 = 结算人数 / 加购人数
支付转化率 = 支付成功人数 / 结算人数
```

分步转化率能帮助我们定位哪一段损失最大。整体转化率下降时，如果不看分步转化率，就像医生只知道病人不舒服，却不知道是发烧、咳嗽还是胃痛。

课堂讲解时可以用一个简单例子：

```text
进入商城 10,000 人
浏览商品 6,000 人
加入购物车 1,800 人
提交结算 1,200 人
支付成功 900 人
```

这时整体转化率是 900 / 10,000 = 9%。但更重要的是分步观察：

- 进入到浏览：60%，说明首页承接还可以。
- 浏览到加购：30%，说明商品吸引力一般。
- 加购到结算：66.7%，说明购物车到结算损失较大。
- 结算到支付：75%，说明支付环节仍有损失但不是最大问题。

如果管理者只问“怎么提升 GMV”，答案会很空。漏斗分析能把问题变具体：优先检查购物车到结算之间的运费、优惠、库存和价格展示。

这就是流量漏斗的商业价值：它把一个大问题拆成几个可行动的小问题。
