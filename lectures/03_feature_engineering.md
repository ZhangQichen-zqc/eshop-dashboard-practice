# 第 3 章 用户建模宽表与特征工程：把“用户价值”变成可计算字段

## 1. 本章导读：模型不懂“高价值用户”

运营同学经常会说：

> 我们要找高价值用户。  
> 我们要找快流失的用户。  
> 我们要找对优惠券敏感的用户。  
> 我们要找最近很活跃但还没购买的用户。

这些话在业务上很自然，但模型听不懂。模型不知道什么叫“高价值”，也不知道什么叫“快流失”。模型只认识字段和数字。

所以我们要做一件事：把业务语言翻译成数据字段。

例如：

| 业务说法 | 数据字段 |
|---|---|
| 高价值用户 | 累计消费金额高 |
| 忠诚用户 | 订单次数多 |
| 最近活跃 | 最近购买天数小，活跃天数多 |
| 可能流失 | 很久没购买 |
| 券敏感用户 | 领券多、用券多 |
| 高意向未转化 | 浏览多、加购多，但订单少 |

这个过程就叫特征工程。

特征工程不是把字段越做越多越好，而是围绕商业问题，构造对决策有意义的字段。

## 2. 本章学习目标

学完本章后，学生应该能够：

1. 解释什么是用户建模宽表。
2. 区分主键、属性字段、行为字段、交易字段。
3. 理解 RFM：Recency、Frequency、Monetary。
4. 理解缺失值为什么不能随便删除。
5. 理解标签和特征的区别。
6. 解释什么是数据泄漏。
7. 构建一张可以支撑复购预测和客户分群的用户宽表。

## 3. 什么是用户建模宽表

宽表可以先用一句话理解：

> 一行代表一个分析对象，一列代表一个可用于分析的特征。

本章的分析对象是用户，所以我们要做的是用户宽表：

```text
一行 = 一个用户
一列 = 一个用户特征
```

一个简化的用户宽表可能长这样：

| user_id | register_channel | member_level | order_count | total_paid | recency_days | event_count | coupons_used |
|---|---|---|---:|---:|---:|---:|---:|
| user_001 | search | gold | 10 | 8000 | 5 | 120 | 3 |
| user_002 | social | normal | 1 | 99 | 300 | 4 | 0 |

模型后续要预测“用户是否复购”，或者判断“用户属于哪类群体”，就可以基于这张表完成。

### 3.1 为什么不是一行一个订单

订单表是一行一个订单。如果一个用户买了 5 次，他在订单表里可能有 5 行。

但复购预测的问题是：

> 这个用户未来会不会复购？

分析对象是用户，不是订单。因此模型样本应该是一行一个用户。

如果不把订单聚合到用户层，模型会混乱：同一个用户的多笔订单可能被当成多个独立样本。

### 3.2 主键是什么

主键是用来唯一识别一行数据的字段。

在用户宽表中，`user_id` 应该是主键。一个 `user_id` 只能出现一次。

如果同一个用户出现多行，就说明宽表没有构造好。后续模型会把一个人当成多个人，分析结果就不可信。

## 4. 特征是什么

特征是模型用来判断的输入信息。

在商业上，特征可以理解为：

> 描述用户过去状态和行为的证据。

例如：

- 用户注册渠道
- 用户会员等级
- 历史订单数
- 累计消费金额
- 最近一次购买距今天数
- 最近行为次数
- 优惠券使用次数

这些字段都可能帮助我们判断用户未来行为。

## 5. 标签是什么

标签是模型要预测的目标。

例如第 4 章复购预测中，标签是：

```text
未来 60 天是否复购
```

如果复购，标签为 1；如果没有复购，标签为 0。

特征和标签必须分清楚。

特征是“预测前已经知道的信息”。标签是“未来要预测的结果”。

如果把未来结果混进特征，就会发生数据泄漏。

## 6. 什么是数据泄漏

数据泄漏是建模中非常严重的问题。它指模型训练时看到了现实预测时不可能提前知道的信息。

举例：

我们要在 2026-01-01 预测未来 60 天是否复购。那么 2026-01-01 之后的订单、浏览、用券，都不能作为特征。因为站在 2026-01-01 那一天，我们还不知道未来发生了什么。

错误做法：

```text
用 2026-02 的购买次数预测 2026-01 到 2026-02 是否复购
```

这就像考试时偷看答案。模型效果会看起来很好，但上线后失效。

正确做法：

```text
只用 2026-01-01 之前的数据构造特征
再用 2026-01-01 之后 60 天是否购买构造标签
```

## 7. RFM：最经典的用户价值框架

RFM 是用户分析中非常经典的框架。它包括三个词：

```text
R = Recency
F = Frequency
M = Monetary
```

### 7.1 Recency：最近一次购买距今天数

Recency 表示用户最近一次购买距离分析日有多久。

如果用户昨天刚买过，Recency 很小；如果用户 300 天没买，Recency 很大。

注意：Recency 是“距离”，不是“分数”。数值越小，通常越活跃。

商业含义：

- Recency 小：用户近期有购买行为，可能仍然活跃。
- Recency 大：用户很久没买，可能沉睡或流失。

运营动作：

- Recency 小且消费高：维护关系，推荐新品。
- Recency 大但历史价值高：召回。
- Recency 大且历史价值低：低成本自动化触达。

### 7.2 Frequency：购买频次

Frequency 表示用户买过多少次。

购买频次高，通常说明用户对平台有信任，也说明用户有持续需求。

商业含义：

- Frequency 高：可能是忠诚用户。
- Frequency 低：可能是新客、一次性购买者或低粘性用户。

### 7.3 Monetary：消费金额

Monetary 表示用户累计消费金额。

消费金额高，说明用户对 GMV 贡献大。但要提醒学生：消费金额高不一定毛利高。如果他买的都是低毛利商品，利润贡献不一定高。

商业含义：

- Monetary 高：高价值用户。
- Monetary 低：普通用户或低价值用户。

## 8. 行为特征：用户看了什么、做了什么

订单只能记录购买结果，但用户购买前会有大量行为。

例如：

- 浏览首页
- 浏览商品页
- 搜索
- 加入购物车
- 进入结算

这些行为记录在 `fact_traffic` 中。

行为特征能帮助我们识别兴趣。

例如：

| 用户行为 | 可能含义 |
|---|---|
| 浏览多、购买少 | 有兴趣但犹豫 |
| 加购多、支付少 | 价格、运费或结算环节有阻碍 |
| 活跃天数多 | 持续关注 |
| 只来一次 | 偶然访问 |

## 9. 优惠券特征：价格敏感度

优惠券数据来自 `fact_coupon_use`。

常见字段：

- `coupons_issued`：发给用户多少券
- `coupons_used`：用户用了多少券
- `is_used`：某张券是否使用

如果一个用户领券多、用券多，说明他可能对价格敏感。

但价格敏感不一定是坏事。如果用户原本不买，优惠券带来增量购买，就是好事；如果用户本来就会买，优惠券只是让他少付钱，就会侵蚀利润。

所以优惠券特征要和 ROI、对照组一起理解。

## 10. 缺失值是什么，为什么不能随便删

缺失值就是数据为空。

用户宽表中常见缺失：

- 某用户没有订单，所以 `order_count` 为空。
- 某用户没有行为，所以 `event_count` 为空。
- 某用户没有优惠券，所以 `coupons_used` 为空。

这些缺失不一定是错误。没有订单，本身就是一种业务信息。

因此我们不能简单删除没有订单的用户，而要根据业务含义填补。

例如：

- 没有订单：`order_count = 0`
- 没有消费：`total_paid = 0`
- 没有行为：`event_count = 0`
- 没有最近购买日期：`recency_days = 999`，表示很久没买或从未购买

## 12. Notebook 代码讲解

### 12.1 读取多张表

```python
users = load_table("dim_user")
orders = paid_orders()
traffic = load_table("fact_traffic")
coupons = load_table("fact_coupon_use")
```

教师讲解：

这里不是简单读取数据，而是把用户系统、交易系统、行为系统、营销系统的数据拿到同一个分析环境里。

### 12.2 构造 RFM

```python
rfm = orders.groupby("user_id").agg(...)
```

业务含义：

按用户汇总订单历史，得到每个用户的消费画像。

### 12.3 构造行为特征

```python
behavior = traffic.groupby("user_id").agg(...)
```

业务含义：

把用户的浏览和操作行为汇总成活跃度。

### 12.4 构造优惠券特征

```python
coupon_feature = coupons.groupby("user_id").agg(...)
```

业务含义：

判断用户是否被优惠券触达，是否真的使用优惠券。

### 12.5 合并宽表

```python
wide = users.merge(rfm).merge(behavior).merge(coupon_feature)
```

业务含义：

把“用户是谁”“买过什么”“看过什么”“用过什么券”放到一行里。

## 16. 教师扩展讲稿：特征工程是把业务经验翻译成模型语言

同学们，第 3 章是很多学生从“会看报表”走向“会建模型”的关键一章。模型本身不会理解业务。模型不知道什么叫“高价值用户”，不知道什么叫“最近很活跃”，也不知道什么叫“价格敏感”。它只能读取一列一列数字或类别字段。

特征工程就是把业务经验翻译成模型能理解的字段。

例如，运营同学说：

```text
最近买过、买得多、经常浏览、愿意用券的用户，更可能复购。
```

模型听不懂这句话。我们必须把它拆成字段：

```text
days_since_last_order
total_gmv
order_count
page_view_count_30d
coupon_use_rate
```

这些字段就是特征。特征工程的质量，决定了模型能看到多少有用信息。

很多时候，模型效果不好，不是算法不够高级，而是特征没有把业务逻辑表达出来。一个普通模型加上好特征，往往比复杂模型加上粗糙特征更有价值。

## 17. 什么是用户宽表

用户宽表是一张以用户为粒度的建模表。它的基本要求是：

```text
一行 = 一个用户
一列 = 一个用户特征或标签
```

为什么要做宽表？因为很多机器学习模型要求每个样本是一行。如果我们要预测用户是否复购，样本单位就是用户，而不是订单、页面浏览或优惠券记录。

原始业务数据通常分散在多张表：

- 用户表记录用户是谁。
- 订单表记录用户买了什么、花了多少钱。
- 流量表记录用户看了什么、点了什么。
- 优惠券表记录用户有没有领券、有没有用券。

这些表的粒度不同。订单表是一人多单，行为表是一人多次浏览，优惠券表是一人多张券。模型不能直接把这些明细表拼在一起，否则一名用户会出现多行，样本就混乱了。

宽表的作用就是把不同粒度的数据统一到用户粒度。

## 18. 主键：为什么 user_id 不能重复

用户宽表最重要的检查之一，是 `user_id` 是否唯一。

如果宽表里同一个用户出现多行，模型就会把同一个人当成多个样本。这会带来两个问题。

第一，训练数据被重复计算。一个活跃用户可能因为订单多而出现很多行，模型会过度重视这类用户。

第二，训练集和测试集可能泄漏。同一个用户的一部分记录进入训练集，另一部分进入测试集，模型看似效果很好，其实是在“认熟人”。

所以，本章参考结果中“用户主键重复数：0”非常重要。它说明这张宽表在样本粒度上是干净的。

## 19. 标签 label：模型要预测的答案

标签是模型要学习的目标。比如第 4 章复购预测中，标签可能是：

```text
未来 30 天是否复购
```

标签必须有清楚的时间窗口。不能模糊地说“是否复购”，因为复购发生在未来多久内会影响样本定义。

如果标签定义不清，模型训练就没有稳定目标。

标签还必须和特征分开时间。特征只能来自预测时点之前，标签来自预测时点之后。如果用未来信息构造特征，就会发生数据泄漏。

例如，要预测用户 7 月是否复购，特征只能使用 6 月及以前的数据，不能使用 7 月浏览或购买信息。

## 20. 数据泄漏：模型考试作弊

数据泄漏是初学者建模中最危险的问题之一。

可以把模型训练想象成考试。特征是复习资料，标签是标准答案。如果你把标准答案的一部分放进复习资料里，学生考试当然高分，但这个分数没有意义。

例子一：预测未来是否复购，却把“未来 30 天订单数”作为特征。这个字段本身已经包含答案。

例子二：预测用户是否流失，却把“流失后是否收到召回短信”作为特征。这个字段发生在预测时点之后。

例子三：训练集和测试集没有按用户或时间正确拆分，导致同一用户信息同时出现在训练和测试中。

数据泄漏会让模型在线下看起来特别好，上线后突然失效。专业分析师必须在特征工程阶段主动检查泄漏风险。

## 21. RFM 特征的商业含义

RFM 是用户建模的基础。

Recency 表示最近一次购买距离现在多久。它回答“用户离我们有多远”。数值越小，通常说明用户越活跃。

Frequency 表示购买频次。它回答“用户买得勤不勤”。频次越高，用户习惯越强。

Monetary 表示消费金额。它回答“用户贡献价值高不高”。金额越高，用户历史价值越大。

RFM 的优势是简单、解释性强。运营人员很容易理解：

- 最近买过、买得多、花得多：核心价值用户。
- 很久没买、过去花得多：高价值沉睡用户。
- 最近买过、金额不高：新客或低客单用户。
- 买得少、花得少、很久没来：低活跃用户。

RFM 不只是建模特征，也可以直接支持用户分层运营。

## 22. 行为特征：购买前的信号

订单数据反映已经发生的购买，行为数据反映购买前的兴趣。

一个用户近期频繁浏览商品、搜索关键词、加入购物车、查看优惠活动，即使还没有购买，也可能处在决策过程中。

常见行为特征包括：

- 最近 7 天浏览次数。
- 最近 30 天商品详情页浏览次数。
- 加购次数。
- 收藏次数。
- 搜索次数。
- 访问天数。
- 活动页点击次数。

行为特征的价值在于提前捕捉意向。只有订单特征时，模型只能看见过去买过的人；加入行为特征后，模型能看见“正在考虑购买”的人。

## 23. 优惠券特征：价格敏感度的线索

优惠券使用情况可以帮助判断用户是否价格敏感。

如果一个用户经常领券但不用券，说明他可能只是被活动吸引，但购买意愿不强。

如果一个用户领券后经常使用，说明优惠对他有驱动力。

如果一个用户几乎不用券也经常购买，说明他可能更看重商品、品牌或便利性，不一定需要高额补贴。

常见优惠券特征包括：

- 领券次数。
- 用券次数。
- 用券率。
- 平均优惠金额。
- 最近一次用券距今天数。
- 用券订单 GMV 占比。

这些特征能支持差异化补贴。对价格敏感用户可以用门槛券促进购买；对本来就会买的用户，不应过度补贴。

## 24. 缺失值不是垃圾

很多学生看到缺失值就想删除。实际上，缺失本身可能有业务含义。

例如，一个用户没有订单记录，不一定是脏数据，他可能是注册未购买用户。这个信息对转化预测很重要。

一个用户没有优惠券使用记录，也不一定要删除。他可能从未领券，也可能领了但没用，这两种情况都值得区分。

处理缺失值时要先问：

```text
缺失是数据错误，还是业务上没有发生？
```

如果是数据错误，要修复或剔除。

如果是业务上没有发生，缺失可以填 0，或者增加一个是否缺失的标记字段。

## 25. 特征粒度和时间窗口

同一个业务含义，用不同时间窗口会产生不同特征。

例如浏览次数可以是：

- 最近 7 天浏览次数。
- 最近 30 天浏览次数。
- 最近 90 天浏览次数。
- 历史累计浏览次数。

最近 7 天反映短期兴趣，最近 90 天反映长期活跃，历史累计反映总体关系。模型可能同时需要这些特征。

时间窗口的选择要服从业务节奏。快消品可能看 7 天、30 天；家电家具可能购买周期长，需要看 90 天、180 天。

不要机械地认为窗口越多越好。窗口太多会增加复杂度，也可能引入噪声。好的特征窗口应该能解释用户决策周期。

## 26. 本章结果的管理层表达

基于真实 ETL，本章用户宽表覆盖 20,000 个用户，其中购买用户 17,507 人，买家率约 87.53%，用户主键重复数为 0，核心特征约 15 个。

管理层表达可以这样写：

```text
我们已将用户、订单、流量和优惠券数据整合为用户级宽表，一行代表一个用户，主键无重复。当前特征覆盖历史消费、近期活跃和优惠券响应，能够支撑后续复购预测和客户分群。下一步建议补充不同时间窗口的行为特征、品类偏好和价格敏感度特征，以提升模型对购买意向和用户价值的识别能力。
```

这段话没有直接讲代码，但清楚说明了宽表的价值：统一建模底座。

## 27. 课堂活动：把一句业务话翻译成特征

教师可以给学生几句业务话，让他们翻译成特征字段。

```text
这个用户最近很活跃。
这个用户很久没买了。
这个用户喜欢打折。
这个用户偏好高客单商品。
这个用户可能正在犹豫购买。
```

学生需要写出字段名、来源表和计算方式。

例如：

```text
业务话：这个用户最近很活跃
字段：page_view_count_7d
来源表：fact_traffic
计算：统计用户最近 7 天浏览事件数
用途：识别短期购买意向
```

这个活动能让学生理解：特征工程不是写代码凑字段，而是把业务判断变成可计算证据。
