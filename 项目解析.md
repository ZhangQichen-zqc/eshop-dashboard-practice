# 项目解析：eshop-dashboard-practice

## 一、一句话说清楚这是什么

**这是一个融合了「IT 项目管理」和「数据挖掘」两门课的期末综合实验。** 老师给你一个现成的电商系统（带模拟数据），你要做的是：在这个系统之上，用 Python 搭建一个数据分析仪表盘，用数据挖掘的方法告诉"老板"——这个电商怎么经营才能更好。

---

## 二、老师到底给了你什么

你拿到的文件夹里有这些东西：

```
eshop-dashboard-practice/
├── client/          ← React 写的商城前台（你基本不需要改）
├── server/          ← Node.js 写的商城后台 + SQLite 数据库（数据来源）
│   ├── src/server.js    ← 商城 API，包含 ETL 只读接口
│   ├── src/db.js        ← 数据库表结构定义（18 张源表 + 14 个分析视图）
│   ├── src/seed.js      ← 数据生成脚本（按固定随机种子生成 20,000 用户、100,000 订单等）
│   └── data/eshop.sqlite ← 这是你的数据金矿（24 个月的电商经营数据）
├── docs/           ← 教学文档（课程大纲、数据字典、作业要求、功能需求）
├── exports/        ← CSV 导出的分析数据（跟 SQLite 内容一样，方便你用 pandas 读）
├── scripts/        ← 开发启动脚本
├── package.json    ← 项目配置（定义了 install:all、dev、verify 等命令）
├── lectures/       ← 你自己放的数据挖掘课程讲义（第 1-9 章）
├── tasks.md        ← 你的任务清单
└── tasks_final.md  ← 课堂演示步骤（相当于老师给的"标准路线图"）
```

**关键理解**：`client/` 和 `server/` 是商城业务系统，它提供了一个真实的电商场景。你真正要写的新代码，是一个全新的 `analytics_dashboard/` 目录（Python + FastAPI），这个目录目前还不存在，**等待你去创建**。

## 三、这个项目要你做什么

### 3.1 宏观目标

你不是要重做商城前台（不用改购物车、下单、支付那些），你要做的是**后台经营分析与决策支持系统**。简单说：

> 用数据挖掘方法，分析这个模拟商城的经营数据，找到经营问题和增长机会，然后把分析结果做成一个老板能看懂的仪表盘。

### 3.2 具体需求（来自 `student-feature-requirements.md`）

老师把工作拆成了 12 个模块（R0 ~ R11），最低要求完成 7 个：

| 模块 | 做什么 | 用到的数据挖掘知识 |
|------|--------|-------------------|
| **R0** 数据接入与清洗 | 从 SQLite/CSV 读数据，检查质量，构建清洗流水线和建模宽表 | 第 2 章 ETL、第 3 章特征工程 |
| **R1** 经营驾驶舱 | GMV、订单数、客单价、毛利、退款率等指标卡 + 趋势图 + 筛选器 | 第 1 章商业指标、第 2 章经营诊断 |
| R2 流量漏斗 | 从访问→浏览→加购→结算→支付的转化分析 | 漏斗分析、渠道对比 |
| **R3** RFM 与用户运营 | 高价值/潜力/沉默用户分群，cohort 留存 | 第 5 章客户分群 |
| **R4** 分类预测模型 | 复购预测 / 流失预警 / 券敏感度 / 退款风险（四选一） | 第 4 章分类算法 |
| R5 聚类分群 | 用户聚类或商品聚类，给每个簇命名和策略 | 第 5 章聚类分析 |
| R6 关联规则 | 购物篮分析，发现"买 A 也会买 B"的组合 | 第 6 章关联规则 |
| **R7** 时间序列预测 | 预测未来 GMV / 销量 / SKU 需求 | 第 7 章时间序列 |
| **R8** 营销活动评估 | 活动 ROAS、优惠券核销率、增量分析 | 第 8 章营销归因 |
| R9 履约售后 | 配送延迟、退款、评论分析 | 多维度分析 |
| R10 库存策略 | 动销率、ABC 分类、补货/清仓建议 | 库存管理 |
| **R11** 综合决策中心 | 把以上所有发现汇总成经营建议和行动计划 | 第 9 章综合决策 |

**最低交付**（加粗的）：R0 + R1 + R3 + **一个分类模型(R4)** + **一个时间序列预测(R7)** + R8 + R11

### 3.3 IT 项目管理部分的要求

- 用 **Git** 管理版本，提交历史要能体现功能演进（环境搭建 → 后端 → 仪表盘 → Docker）
- 用 **Docker Compose** 把整个系统部署到 Ubuntu Linux
- 能解释每个指标对应的数据挖掘知识点

## 五、老师为什么给你这个文件夹结构

这其实是按照软件工程项目的真实流程设计的：

1. **先有业务系统**（client + server = 商城），这样才能产生数据
2. **然后才有分析系统**（analytics_dashboard = 你要新建的），这是数据挖掘落地的地方
3. **docs/ 和 exports/** 让你可以从不同入口（直连数据库 vs 读 CSV）接入数据，理解 ETL 的概念
4. **这就是实际工作中的"混合编程"场景**：旧系统是 Node.js，新分析服务用 Python，两者通过数据库和 API 协作，而不是重写一切

老师的设计思路是：**模拟真实企业的数据分析团队工作流程**——业务系统已经在跑了，分析团队接到任务：用数据告诉管理层这个业务到底怎么样，下一步该做什么。

## 六、数据挖掘如何落地到这个项目中

你的讲义（lectures/）和这个项目的对应关系：

| 你的讲义 | 在这个项目中怎么用 |
|----------|-------------------|
| 第 1 章 商业问题定义与 ETL | 理解 GMV、毛利、转化率等指标；用 `/api/etl/*` 接口或 CSV 读数据 |
| 第 2 章 经营健康诊断 | 实现 R1 经营驾驶舱——指标拆解、趋势分析、结构分析 |
| 第 3 章 特征工程 | 实现 R0 中的用户建模宽表——把"高价值用户"翻译成 RFM 字段 |
| 第 4 章 复购预测 | 实现 R4 分类模型——观察窗口/预测窗口/模型评估/阈值选择 |
| 第 5 章 客户分群 | 实现 R5 聚类——K-Means 分群、轮廓系数、簇画像命名 |
| 第 6 章 关联规则 | 实现 R6 购物篮分析——支持度/置信度/提升度 |
| 第 7 章 销售预测 | 实现 R7 时间序列——移动平均、季节性分解、安全库存 |
| 第 8 章 营销归因 | 实现 R8 活动评估——ROAS、增量分析、预算建议 |
| 第 9 章 综合项目 | 实现 R11 决策中心——把模型结果变成经营建议 |

**数据挖掘不是孤立跑模型**，而是：**业务问题 → 选择数据 → 特征工程 → 建模 → 评估 → 落地为运营动作**。这正是 CRISP-DM 方法论的核心。

## 七、建议的起步顺序

如果你想从完全迷茫到能上手，按这个顺序来：

1. **先跑起来看看**：执行 `npm run install:all` 和 `npm run dev`，看商城能不能在浏览器打开
2. **理解数据**：读 `docs/data-dictionary.md`，打开 `server/data/eshop.sqlite` 用 SQLite 浏览器看看表里有什么
3. **读作业要求**：`docs/student-feature-requirements.md` 告诉你每个模块的验收标准
4. **从 R1 开始做**：经营驾驶舱是最容易出效果的第一步——先算出 GMV、订单数、客单价，做出第一张图表
5. **然后按难度递增**：R3(RFM) → R4(分类) → R7(预测) → 其他模块
6. **每做完一个模块 commit 一次**：这样 Git 历史能体现功能演进

## 八、关键数据表速查

你大部分分析用这些分析视图就够了，不需要直接查源表：

| 表名 | 一句话说明 |
|------|-----------|
| `dim_date` | 日期维度（年份、月份、是否周末） |
| `dim_user` | 用户维度（地区、渠道、会员等级） |
| `dim_product` | 商品维度（类目、品牌、价格带、成本、供应商） |
| `dim_campaign` | 活动维度（渠道、类型、有无对照组、预算） |
| `fact_order` | 订单事实（GMV、客单价、复购分析用的） |
| `fact_order_item` | 订单行事实（品类贡献、毛利、购物篮分析用的） |
| `fact_traffic` | 流量事实（漏斗转化、渠道质量用的） |
| `fact_coupon_use` | 优惠券事实（核销率、券敏感度用的） |
| `fact_refund` | 退款事实（退款率、售后风险用的） |
| `fact_fulfillment` | 履约事实（配送时效、延迟率用的） |
| `fact_inventory_movement` | 库存流水（动销率、缺货风险用的） |
| `fact_product_review` | 评论事实（评分、差评分析用的） |
| `fact_ads_spend` | 广告花费（ROAS、渠道效率用的） |
| `daily_business_summary` | 经营日报（最方便，日期+渠道粒度，直接用） |

## 九、总结

用大白话说：

> 老师给了你一个模拟电商和一堆经营数据，要你扮演数据分析师的角色，用 Python 写一个后台系统，通过数据挖掘的方法分析这个电商的经营状况，然后给"老板"提出具体的经营建议。同时要用 Git 管好代码，最后用 Docker 部署起来。这个过程锻炼的是"从数据到决策"的完整能力，既有技术（Python、FastAPI、机器学习），也有管理（Git 版本控制、Docker 部署、项目规划），还有商业思维（数据分析→经营建议→行动计划）。
